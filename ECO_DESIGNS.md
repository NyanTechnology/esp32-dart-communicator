# 😺 NyanEye 智能低功耗与无线唤醒架构设计书 (REQ-8 Blueprint)

本设计文档旨在应对 NyanEye 智能双屏摆件在**无物理开机按键（插电即开机）以及不可完全切断网络（否则失联无法唤醒）**的核心硬件约束下，如何实现优雅、高效、极致省电的 **`REQ-8` 深度待机与节电休眠模式**。

文档系统性地提出了三档不同梯度的「无线唤醒与阶梯能效管理方案」，作为下一阶段研发实施的技术蓝图。

---

## 🔌 1. 核心硬件约束与能耗杀手分析

在没有物理按键拉低 LDO 使能管脚（`PIN_PWR_EN`）进行硬关机的前提下，设备的“待机”必须在软件与射频协议栈层面进行精细化控制。硬件核心耗电项如下：

*   **双 GC9D01 圆形屏幕及背光**：背光常亮约 `40mA - 80mA`。待机时必须通过 GPIO 拉低彻底熄灭背光，并向 LCD 驱动芯片下发寄存器指令 **`0x10` (Sleep In)**，使屏幕像素偏振和电荷泵进入微安级休眠。
*   **ESP32-C3 射频（Wi-Fi / BLE）**：数据收发瞬时电流达 `100mA - 200mA`。待机时必须依靠分时休眠或跨协议唤醒来降低功耗。

## 🚀 2. 三大待机与无线唤醒设计方案

### 🟢 方案 A：蓝牙辅助唤醒机制（BLE-Assisted Wake-on-LAN）
**「极致省电，App 跨协议门铃唤醒」**
*   **适用场景**：使用纯电池供电、要求超长待机续航、且不介意唤醒时有 2 秒左右连网延迟的场景。

```text
待机状态：
[屏幕完全关闭 & GC9D01 硬件休眠] ──► [Wi-Fi 射频彻底关闭] ──► [CPU 进入 Light Sleep] ──► [仅保留 BLE 低频慢广播 (1.5mA)]
  │
当用户在手机 App 打开局域网/蓝牙面板：
  ▼
手机通过蓝牙连接 C3 并发送 "WAKE_UP" 信号 ──► [C3 被蓝牙中断唤醒] ──► [启动 Wi-Fi 联网] ──► [点亮屏幕] ──► 恢复工作
```

*   **技术细节**：
    1.  **待机触发**：收到待机指令后，关闭 LCD 背光并使能屏幕 `Sleep In` 保护。彻底关闭 Wi-Fi 射频。
    2.  **BLE 降频广播**：将 NimBLE 的广播间隔（Advertising Interval）调整为较慢的 **`1200ms`**（大幅减少射频发射频次）。
    3.  **CPU 休眠**：调用 ESP-IDF 的 `esp_light_sleep_start()` 让 CPU 时钟挂起。此时，蓝牙射频芯片通过硬件定时器定期在广播窗口醒来发送信号。
    4.  **唤醒逻辑**：用户打开 App 扫描到设备的蓝牙慢广播，点击连接并向 `beb5483e-36e1-4688-b7f5-ea07361b26a9`（FFE0 户外控制特征）写入 `"WAKE_UP"`。蓝牙硬件产生中断唤醒 CPU，CPU 立刻重新初始化 Wi-Fi、连网、点亮双屏。
*   **能耗指标**：待机电流约 **`1.5 mA`**（1000mAh 电池可待机约 **30 天**）。

---

### 🟡 方案 B：Wi-Fi 维持轻睡眠唤醒机制（Wi-Fi Connected Light Sleep）
**「随时秒开，局域网一击即中」**
*   **适用场景**：插着 USB 数据线供电，或者电池容量极大，要求在手机局域网面板中拥有“无感秒级唤醒”极致体验的场景。

```text
待机状态：
[屏幕完全关闭 & GC9D01 硬件休眠] ──► [CPU 进入 Light Sleep] ──► [Wi-Fi 维持与路由器的 DTIM 连接 (3.5mA)]
  │
当用户在 App 局域网面板发送任何指令（如 /api/ping）：
  ▼
路由器将 TCP 握手数据包发给 C3 ──► [Wi-Fi 射频硬件中断唤醒 CPU] ──► [亮屏显示] ──► 50毫秒内无感唤醒
```

*   **技术细节**：
    1.  **待机触发**：关闭背光与 LCD 驱动芯片（Sleep In）。
    2.  **开启 Wi-Fi 阶梯省电**：调用 **`esp_wifi_set_ps(WIFI_PS_MAX_MODEM)`**。设备并不与路由器断开连接，而是与路由器达成时钟对齐。
    3.  **DTIM 周期睡眠**：芯片根据路由器的 DTIM 广播周期（通常为 `100ms - 300ms`），在无信号传输间期调用 `esp_light_sleep_start()` 让 CPU 自动挂起休眠，在 DTIM 窗口自动醒来监听寻呼。
    4.  **唤醒逻辑**：当手机在局域网内发起任何 TCP 连接（如 HTTP GET /api/ping 或者是 /api/eyes/apply），无线射频芯片侦测到数据包后，产生硬件级中断瞬间拉醒 CPU。CPU 在 50ms 内亮屏并响应，完成无感唤醒。
*   **能耗指标**：待机平均电流约 **`3.5 mA`**（1000mAh 电池可待机约 **12 天**）。

---

### 🔴 方案 C：脉冲分时连网唤醒机制（Pulse Duty Cycle Deep Sleep）
**「变态续航，工业级断续轮询」**
*   **适用场景**：对功耗极其苛刻，可以接受 1 分钟左右的控制延迟，追求极致单次充电续航的极客场景。

```text
待机状态 (90% 时间)：
[屏幕关闭] ──► [所有无线射频完全断电] ──► [进入 Deep Sleep 极低功耗 (20μA)]
  │  (每隔 60 秒，内部硬件 RTC 定时器溢出唤醒)
  ▼
脉冲监听状态 (10% 时间)：
[启动 BLE 慢广播并监听 2 秒]
  ├──► 手机 App 没在线 ────► [继续深睡眠 Deep Sleep 60秒]
  └──► 手机 App 下发唤醒 ──► [启动 Wi-Fi 并彻底开机复活]
```

*   **技术细节**：
    1.  **待机触发**：设备彻底切断 CPU、内存、射频的所有供电，进入 **`esp_deep_sleep_start()`**（深睡眠模式）。
    2.  **定时唤醒**：配置 RTC 硬件定时器定时时间为 `60秒`。
    3.  **短脉冲监听**：每 60 秒，RTC 唤醒系统进行一次微小的“冷启动”。系统启动后检测到是从睡眠状态唤醒，它**只开启 BLE 慢广播，静默监听 2 秒钟**，完全不开启高耗电的 Wi-Fi。
    4.  **唤醒逻辑**：如果在 2 秒脉冲期内，手机 App（处于打开状态）成功连上蓝牙并发送了唤醒信号，设备彻底启动。如果 2 秒内无任何请求，设备立刻重新进入 Deep Sleep 60 秒。
*   **能耗指标**：待机平均电流低于 **`25 μA`**（1000mAh 电池可保存待机 **数年以上**，几乎等同于自然自放电率）。

## 🏆 3. 头脑风暴架构验收与选择标准

| 指标维度 | 方案 A：蓝牙辅助唤醒 | 方案 B：Wi-Fi 维持轻睡眠 | 方案 C：脉冲深睡眠 |
| :--- | :--- | :--- | :--- |
| **待机电流** | `1.5 mA` (优秀) | `3.5 mA` (良好) | **`20 μA` (极致)** |
| **1000mAh续航** | 约 **30 天** | 约 **12 天** | **数年以上** |
| **唤醒延迟** | 约 `2秒`（需重新连 Wi-Fi） | **< 50ms（零延迟，瞬间无感）** | 约 `0 - 60秒` (取决于轮询周期) |
| **唤醒通道** | BLE 长波广播 ➡️ 写入唤醒 | 局域网任意 TCP/HTTP 请求一击即中 | BLE 2秒超短脉冲窗口 |
| **网络在线状态** | 待机时断网，唤醒后重连 | **待机时依然保持网络关联** | 待机时彻底断网 |
| **代码实现复杂度** | 中等（需处理蓝牙到 Wi-Fi 的状态转换） | **最简单（ESP-IDF 原生驱动支持）** | 较高（需处理 RTC 唤醒冷启动参数判定） |

### 🎯 下阶段开发推荐策略：
*   **第一优先级推荐**：**「方案 B（Wi-Fi Connected Light Sleep）」**。
    *   **理由**：无物理按键设备的核心是 **“不能让用户觉得卡顿和失控”**。方案 B 支持随时随地通过局域网“一击秒醒”，体验丝滑无感。且 `3.5mA` 的电流在日常桌面插电或中等锂电池下表现已经极其优异。
*   **第二优先级推荐（作为可选省电开关）**：**「方案 A（蓝牙辅助唤醒）」**。
    *   若用户在 App 里勾选了“极致省电待机”，则可以降级切入方案 A，用蓝牙慢广播充当功耗极低的“无线门铃”。
